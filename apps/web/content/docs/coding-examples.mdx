## Understand A Codebase

Use this when joining an unfamiliar repository or returning to a project after a
long break.

```text
Read this repository and explain the main runtime path for a user message.
Start from the entrypoint, follow the important modules, and point me to the
files I should read first. Do not edit files.
```

Good follow-up prompts:

- `Trace how configuration is loaded and merged.`
- `Explain how provider selection reaches the model request code.`
- `Find the tests that protect this behavior.`

Expected result:

- A concrete file-by-file map.
- The main control flow in plain language.
- Gaps or confusing areas called out separately from confirmed behavior.

## Fix A Bug

Use this when you can describe the broken behavior and want Devo to make the
smallest useful change.

```text
The docs page tree renders, but one page is missing from the sidebar.
Find the cause, implement the fix, and run the relevant validation commands.
Keep the change scoped to the docs app.
```

Good follow-up prompts:

- `Show me the exact files changed and why.`
- `Add a regression test if this codebase has a local pattern for it.`
- `Run the narrowest check that proves this bug is fixed.`

Expected result:

- A diagnosis tied to code or configuration.
- A scoped patch.
- Validation output summarized clearly.

## Research Experiment

Use this when you want to compare options before committing to a design or
implementation.

```text
Experiment with two ways to represent provider wire API support in the docs.
Prototype both approaches, compare readability and maintenance cost, then keep
the better one. Remove the unused experiment before finishing.
```

Good follow-up prompts:

- `Compare the tradeoffs in a short table.`
- `Keep the experiment isolated so it is easy to discard.`
- `Do not change product behavior until the conclusion is clear.`

Expected result:

- A reversible exploration.
- A short comparison of the options.
- One final implementation, with temporary work cleaned up.

## Prompting Tips

Be explicit about the boundary of the work:

- Say whether Devo should edit files or only inspect.
- Name the package, directory, or feature area when you know it.
- Ask for the validation command you care about.
- Tell Devo when to stop and report instead of broadening the task.
