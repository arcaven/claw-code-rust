## 理解代码库

当你进入一个陌生仓库，或隔了一段时间回到项目时，可以使用这个示例。

```text
阅读这个仓库，并解释用户消息进入系统后的主要运行路径。
从入口开始，跟踪重要模块，并指出我应该优先阅读哪些文件。
不要编辑文件。
```

适合继续追问：

- `跟踪配置是如何加载和合并的。`
- `解释提供方选择如何进入模型请求代码。`
- `找到保护这个行为的测试。`

预期结果：

- 具体的文件阅读地图。
- 用清晰语言描述的主控制流。
- 把不确定或混乱的地方与已确认行为分开说明。

## 修复 Bug

当你能描述损坏行为，并希望 Devo 做最小可用修改时，可以使用这个示例。

```text
文档页面树可以渲染，但侧边栏缺少一个页面。
找到原因，实现修复，并运行相关验证命令。
把改动范围限制在 docs app。
```

适合继续追问：

- `告诉我具体改了哪些文件以及原因。`
- `如果这个代码库已有本地模式，请添加回归测试。`
- `运行能证明此 bug 已修复的最窄检查。`

预期结果：

- 绑定到代码或配置的诊断。
- 范围明确的补丁。
- 清晰总结的验证结果。

## 研究实验

当你想在承诺设计或实现之前比较多个方案时，可以使用这个示例。

```text
实验两种在文档中表达 provider wire API support 的方式。
原型化两个方案，比较可读性和维护成本，然后保留更好的一个。
完成前删除未使用的实验代码。
```

适合继续追问：

- `用一个简短表格比较权衡。`
- `让实验保持隔离，便于丢弃。`
- `结论明确之前不要改变产品行为。`

预期结果：

- 可回滚的探索。
- 对方案的简短比较。
- 一个最终实现，并清理临时工作。

## 提示词建议

明确任务边界：

- 说明 Devo 应该编辑文件还是只检查。
- 如果知道，请指出 package、目录或功能区域。
- 指定你关心的验证命令。
- 告诉 Devo 什么时候应该停止并报告，而不是扩大任务范围。
